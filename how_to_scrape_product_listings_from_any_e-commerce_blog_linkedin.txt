Most e-commerce scrapers fail not because of bad code, but because they were built for one site, one day, one CSS class that no longer exists.

That is the core problem with how most people approach this. They write a script, it works, they move on. Then the site updates its layout, or deploys Cloudflare, or switches to a React storefront, and the whole thing silently falls apart.

Building a scraper that actually holds up across sites requires thinking through a few things most tutorials skip: how to handle JS-rendered product grids that return empty HTML to a plain requests call, how to deal with anti-bot systems that fingerprint headless browsers regardless of what User-Agent you set, and how to manage pagination across sites that use infinite scroll instead of URL parameters.

There is also the selector fragility problem. Class names change. Sites A/B test layouts. A scraper that works today can fail quietly next week. Defensive parsing with fallbacks is not optional for anything you want to run on a schedule.

The post I linked below walks through all of it in detail: static HTML scraping, Playwright for JS-heavy storefronts, anti-bot handling, pagination, AI-based extraction for multi-site pipelines, and production considerations like deduplication and atomic CSV writes.

If you are building anything that needs to pull product data reliably, not just once, it is worth the read.
